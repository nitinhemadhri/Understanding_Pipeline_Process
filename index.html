Fixed 60s Interval for ALL 843 Tables

ONE bell rings every 60s

→ No per-table countdown needed
→ Simple, clean, zero overhead

Layer 01 POLLER pipeline.py

MES DB → Queue 1

Every 60 seconds, poll 843 tables
Uses ThreadPool(8 workers) — all tables in parallel
Fast, non-blocking → dumps rows into queue_1
Does NOT wait for PostgreSQL or Kafka

→ Just reads and hands off. That's it.

queue_1 · in-memory buffer · fast handoff
Layer 02 PG WRITER pg_writer.py

Queue 1 → PostgreSQL raw_data

Runs continuously, reads from queue_1
Batch inserts into raw_data table
Sets push_status = 'PENDING_KAFKA'
Dumps row IDs into queue_2

→ Acts as durable buffer. If Kafka dies, data is safe in PG.

queue_2 · Kafka-ready buffer
Layer 03 KAFKA PROD kafka_producer.py

Queue 2 → Kafka Topic

Reads from queue_2, sends to Kafka topic
Runs continuously, independent of poller
If Kafka is slow → queue_2 buffers the backlog
Sets push_status = 'KAFKA_SENT'

→ Decoupled. Poller never waits for Kafka.

Kafka Topic → consumer reads
Layer 04 S3 WRITER kafka_consumer.py

Kafka Consumer → S3 → Offset Update

Reads from Kafka, writes parquet files to S3
On S3 confirm → sets push_status = 'S3_DONE'
ONLY NOW updates offset_tracker.last_id
This is the source of truth for "what's done"

→ Offset only moves when data lands safely on S3.

❌ THE RACE CONDITION — WITHOUT 3-LAYER DESIGN
T=0s Poll fires. Read rows from MES where ID > last_id=5180000
T=2s 843 tables polled. 15,000 new rows collected.
T=5s Sending to Kafka... Kafka is under load. Processing slowly.
T=60s ⚠️ Next poll fires! Offset NOT updated yet (Kafka still processing).
T=62s Reads SAME rows again → WHERE ID > 5180000 (unchanged!)
T=65s 💥 Duplicate rows sent to Kafka → S3 gets duplicates!

✅ HOW THE 3-LAYER DESIGN SOLVES THIS

T=0s Poll fires → reads from MES WHERE ID > last_id=5180000
T=2s Rows dropped into queue_1. Poller's job is DONE.
T=3s PG Writer reads queue_1 → inserts into raw_data with push_status='PENDING_KAFKA'
T=60s Next poll fires. Reads MES again WHERE ID > last_id (still 5180000)
T=62s New rows from MES land in raw_data as PENDING_KAFKA
T=90s Kafka producer reads PENDING_KAFKA rows from PG. Sends them.
T=95s S3 confirms. push_status → 'S3_DONE'. offset_tracker.last_id → 5180302
T=120s Next poll: WHERE ID > 5180302 ✓ No duplicates.
-- Kafka producer NEVER reads from MES directly
-- It reads from PostgreSQL where push_status = 'PENDING_KAFKA'

WHERE push_status = 'PENDING_KAFKA'
ORDER BY ingested_at ASC
LIMIT 1000;

raw_data Table Schema

  payload       JSONB,
  ingested_at   TIMESTAMPTZ DEFAULT NOW(),
  push_status   TEXT DEFAULT 'PENDING_KAFKA' -- ← state machine
